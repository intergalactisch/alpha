# Intent, Outcome, Request

Status: conceptual core.

This document defines the current Alpha triad:

> Intent points.

> Outcome gives it shape.

> Request asks an actor or capability to help.

This is not a rigid pipeline.

Life is connected. One Intent may shape many Outcomes. Several Intents may converge into one Outcome. One Request may serve multiple Outcomes. A Request may create new context, reveal a boundary, activate memory, produce artifacts, start a process, or show that the original Intent should be revised.

The point of the triad is not to simplify life too much.

The point is to prevent Alpha from collapsing everything into one vague task.

## Why This Matters

Old software often starts with an interface.

Modern AI tools often start with a prompt.

Alpha should start deeper:

- What is being pursued?
- What would make it meaningfully true enough?
- What could be asked of an actor, capability, system, community, service, or external world, if anything?

These are different responsibilities.

If Alpha confuses them, it becomes careless:

- Intent becomes a command.
- Outcome becomes task completion.
- Request becomes commerce, automation, or pressure.

Alpha needs all three, but they must not own each other's job.

## 1. Intent

Intent is sourced direction.

It can come from:

- a person
- a group
- an agent
- a sub-agent
- a process
- a boundary
- a context signal
- a memory representation
- a non-event
- a prior Outcome
- an external Alpha

Intent is not always spoken.

Someone may say:

> "I want this week to feel calmer."

But Alpha may also detect:

> This process has been waiting for three days.

Or an agent may form a derived operational intent:

> Investigate why this plan is blocked.

Those are not equal.

Human and group Intent can be sovereign.

Agent Intent can only be derived, scoped, explainable, interruptible, and tied to an identifiable agent actor.

Alpha may interpret Intent, represent Intent, exchange Intent, connect Intent, and make Intent handleable.

Alpha does not own Intent.

Canonical line:

> Intent is sourced direction; Alpha can interpret it, but does not own it.

## Intent Requirements

Every Intent representation should be able to name:

- source actor
- authority actor or authority basis
- scope
- lineage
- current status
- boundary constraints
- uncertainty

No agent-originated Intent should be attributed to a generic `Agent` when a specific agent or sub-agent initiated it.

Every agent and sub-agent needs a scoped identifier.

This matters because responsibility, transparency, rollback, and trust depend on knowing where Intent came from.

## 2. Outcome

Outcome is what would make Intent meaningfully true enough in context.

Intent points.

Outcome gives it shape.

Outcome is not merely:

- task completion
- engagement
- user satisfaction
- a generated artifact
- an action performed
- a metric improved
- what Alpha guessed

Outcome should be inspectable and editable by the relevant authority.

It should be allowed to change when context changes.

It should remain connected to Intent, boundaries, memory, process, artifacts, and care.

Canonical line:

> Outcome gives Intent shape.

## Outcome Requirements

Outcome may include:

- success meaning
- current status
- enoughness signals
- care requirements
- boundary constraints
- relevant context
- participants
- time horizon
- uncertainty
- correction paths
- pause, revision, fulfillment, and abandonment paths

An Outcome can be:

- proposed
- clarified
- accepted
- active
- blocked
- revised
- fulfilled
- obsolete
- abandoned

Those are statuses.

They should not become the concept name.

Use:

- **Outcome** for the core concept.
- **Outcome State** for current condition.
- **Outcome Artifact** for a handleable representation.
- **Outcome View** for a scoped rendering or query perspective.
- **Outcome Event** for meaningful change.
- **Outcome Protocol** if exchange rules become necessary.

## 3. Request

Request is how Alpha asks an actor or capability to help shape an Outcome.

Request is not the same as demand.

Demand can sound like markets, pressure, extraction, and supply-demand logic.

Request is broader and more humane.

A Request may ask for:

- internal agent work
- tool use
- capability execution
- creation
- coordination
- modification
- sourcing
- answering
- preparation
- refusal
- negotiation
- verification
- translation
- scheduling
- help
- silence
- no action

Requests can be internal or external.

An internal Request may ask an Alpha agent, sub-agent, tool, process, memory capability, Life Graph query, or local service to help.

An external Request may ask a person, group, business, creator, institution, service, device, robot, marketplace, community, or another Alpha to help.

Request is how Alpha moves beyond finite catalogs without turning every need into advertising, shopping, or commerce.

Canonical line:

> Request is how Alpha asks an actor or capability to help shape an Outcome.

## Request Requirements

Request may include:

- linked Intent
- linked Outcome
- source actor
- authority basis
- constraints
- budget
- time horizon
- quality bar
- acceptable fulfillment types
- sustainability preference
- privacy constraints
- safety requirements
- human approval requirements
- refusal conditions
- provider or capability constraints
- provenance requirements

Request should remain subordinate to Intent and Outcome.

It must not become hidden advertising, forced commerce, or autonomous fulfillment pressure.

Use:

- **Request** for the core concept.
- **Request State** for current condition.
- **Request Artifact** for a handleable representation.
- **Request View** for inspection, editing, and choice.
- **Request Event** for meaningful change.
- **Request Protocol** for exchange.

## The Triad

The simplest form:

> Intent -> Outcome -> Request

But Alpha should understand the richer reality:

```text
Intent asks: What is being pursued?
Outcome asks: What would make it meaningfully true enough in context?
Request asks: What could an actor or capability do, provide, create, coordinate, answer, decline, or transform?
```

This can produce:

- no visible output
- a clarification
- an artifact
- a process
- a message
- a media moment
- a business workflow
- a learning path
- a social coordination
- a purchase plan
- a creator commission
- a service request
- a public contribution
- a private hold
- a refusal
- a pause
- a change in context

Request is optional.

Not every Intent needs a Request.

Not every Outcome should ask an actor or capability for something.

Sometimes the right move is to rest, wait, remember, talk, delete, decline, or do nothing.

## Agent-Originated Intent

Agents and sub-agents may form operational Intent.

Examples:

- A researcher agent intends to compare options.
- A boundary agent intends to check whether private memory may be used.
- A process agent intends to resume a waiting task.
- A recovery agent intends to diagnose a failed tool call.
- A context agent intends to inspect whether an old memory is relevant.

This is allowed only if:

- the agent has a scoped identifier
- the Intent is derived or delegated
- the Intent has authority scope
- the Intent has lineage
- the Intent is explainable
- the Intent is interruptible
- the Intent respects the Boundary Layer
- the Intent does not pretend to be human desire

Agent Intent is operational.

Human and group Intent can be sovereign.

## Relationship To Other Alpha Parts

### Life Graph

The Life Graph connects Intent, Outcome, Request, artifacts, memory representations, processes, boundaries, people, agents, tools, and events.

It should answer:

- Which Intent shaped this Outcome?
- Which Requests came from this Outcome?
- Which agent initiated this derived Intent?
- Which boundary limited this Request?
- Which artifact made this Outcome inspectable?
- Which process is currently blocked?
- Which memory representation changed the context?

The Life Graph should not own Intent, Outcome, or Request.

It connects them.

### Boundary Layer

The Boundary Layer governs what Alpha may, may not, and must ask before doing.

Intent can be constrained by boundaries.

Outcome can require boundary checks.

Request can require explicit approval.

High-consequence Requests should not execute without the right authority.

### Artifact

Artifacts make parts of Intent, Outcome, or Request handleable.

Examples:

- Intent Artifact
- Outcome Artifact
- Request Artifact
- Request View
- Outcome View

Artifacts are handles.

They are not the living Intent, whole Outcome, or complete Request system.

### Process Layer

The Process Layer tracks dynamic work:

- waiting
- queued
- running
- blocked
- sleeping
- revised
- completed
- abandoned

Requests often create processes.

Processes may generate new derived Intent.

Processes may update Outcome State.

### Alignment Layer

Alignment checks whether Alpha's interpretation, Outcome, Request, artifact, process, or action still fits:

- Intent
- context
- boundaries
- care
- authority
- uncertainty
- user correction
- current reality

Alignment prevents Request from running away from Intent.

## Failure Modes

### Intent Collapse

Alpha treats Intent as a literal command.

Risk:

> The deeper meaning disappears.

Correction:

> Preserve source, authority, scope, uncertainty, and context.

### Outcome Collapse

Alpha treats Outcome as task completion.

Risk:

> Alpha finishes something that did not actually help.

Correction:

> Keep Outcome inspectable, editable, and connected to enoughness.

### Request Collapse

Alpha treats Request as commerce or automation.

Risk:

> Alpha becomes shopping, advertising, outsourcing, or pressure.

Correction:

> Keep Request subordinate to Intent, Outcome, boundaries, and care.

### Agent Blur

Alpha attributes all AI-originated work to one vague agent.

Risk:

> Responsibility, source, authority, and rollback become unclear.

Correction:

> Every agent and sub-agent has a scoped identifier and lineage.
