# Intent, Align, Outcome

Status: conceptual core.

This document defines the current Alpha core movement:

> Intent points.

> Align creates shared understanding.

> Outcome names what would be true enough.

This movement happens inside a Thread.

> Thread is the organizing reference for the movement from Intent through Align to Outcome.

This is not a rigid pipeline.

Life is connected. One Intent may lead to many Outcomes. Several Intents may converge into one Outcome. Align may involve conversation, context, memory, boundaries, uncertainty, agents, sub-agents, tools, artifacts, waiting, silence, or doing nothing.

The point is not to over-model life.

The point is to prevent Alpha from collapsing everything into one vague task.

See also: [Alpha Threads](58-alpha-threads.md).

## Why This Matters

Old software often starts with an interface.

Modern AI tools often start with a prompt.

Alpha should start deeper:

- What is being meant or pursued?
- Does Alpha understand it correctly, in this context?
- What would make it meaningfully true enough?

These are different responsibilities.

If Alpha confuses them, it becomes careless:

- Intent becomes a command.
- Align becomes hidden assumption.
- Outcome becomes task completion.

Alpha needs to keep the movement flat enough to work and clear enough to trust.

## 1. Intent

Intent is sourced direction.

It can come from:

- a person
- a group
- an agent
- a sub-agent
- a process
- a boundary
- a context signal
- a memory representation
- a non-event
- a prior Outcome
- an external Alpha

Intent is not always spoken.

Someone may say:

> "I want this week to feel calmer."

But Alpha may also detect:

> This process has been waiting for three days.

Or an agent may form a derived operational intent:

> Investigate why this plan is blocked.

Those are not equal.

Human and group Intent can be sovereign.

Agent Intent can only be derived, scoped, explainable, interruptible, and tied to an identifiable agent actor.

Alpha may interpret Intent, represent Intent, exchange Intent, connect Intent, and make Intent handleable.

Alpha does not own Intent.

Canonical line:

> Intent is sourced direction; Alpha can interpret it, but does not own it.

## Intent Requirements

Every Intent representation should be able to name:

- source actor
- authority actor or authority basis
- scope
- lineage
- current status
- boundary constraints
- uncertainty

No agent-originated Intent should be attributed to a generic `Agent` when a specific agent or sub-agent initiated it.

Every agent and sub-agent needs a scoped identifier.

This matters because responsibility, transparency, rollback, and trust depend on knowing where Intent came from.

## 2. Align

Align is the act of reaching shared understanding between Alpha and the relevant authority.

That authority may be:

- a person
- a group
- a guardian
- a business owner
- a community
- an agent acting under delegated scope
- another Alpha acting through a safe envelope

Align is not a heavy object.

It is not a bureaucracy.

It is the moment Alpha asks, implicitly or explicitly:

> Do I understand what is meant, in this context, with these boundaries and uncertainties?

Align may include:

- listening
- reflecting
- clarifying
- checking assumptions
- sensing context
- checking boundaries
- exposing uncertainty
- asking a small question
- comparing interpretations
- showing options
- waiting
- accepting correction
- refusing unsafe direction
- doing nothing

Align can be quiet.

Align can involve agents.

Align can also be one sentence.

It should stay as flat as possible until consequence requires more structure.

Canonical line:

> Align turns Intent into shared understanding.

Align includes the important turns in understanding that Alpha may need to explain later.

Align should not preserve every thought, token, tool call, or internal agent step.

It may preserve meaningful movement: a correction, accepted interpretation, surfaced uncertainty, boundary check, context bridge, relevant agent contribution, or change that helps explain how Outcome became possible.

## Align Requirements

Align should preserve:

- source
- authority
- context
- boundaries
- uncertainty
- user correction
- agent identity when agents are involved
- cost awareness
- attention respect
- reversibility where possible

Align should not force every internal movement into a formal object.

Not every agent activation needs a named artifact.

Not every tool call needs to become a process.

Not every thought needs to become visible.

Alpha should only add structure when it helps trust, control, safety, memory, coordination, or continuation.

## 3. Outcome

Outcome is the aligned understanding of what would make Intent meaningfully true enough in context.

Intent points.

Align creates shared understanding.

Outcome names what would be true enough.

Outcome is not merely:

- task completion
- execution
- engagement
- user satisfaction
- a generated artifact
- an action performed
- a metric improved
- what Alpha guessed

Outcome should be inspectable and editable by the relevant authority.

It should be allowed to change when context changes.

It should remain connected to Intent, boundaries, memory, process, artifacts, and care.

Canonical line:

> Outcome is aligned understanding of meaningful enoughness.

## Outcome Requirements

Outcome may include:

- success meaning
- current status
- enoughness signals
- care requirements
- boundary constraints
- relevant context
- participants
- time horizon
- uncertainty
- correction paths
- pause, revision, fulfillment, and abandonment paths

An Outcome can be:

- proposed
- clarified
- accepted
- active
- blocked
- revised
- fulfilled
- obsolete
- abandoned

Those are statuses.

They should not become the concept name.

Use:

- **Outcome** for the core concept.
- **Outcome State** for current condition.
- **Outcome Artifact** for a handleable representation.
- **Outcome View** for a scoped rendering or query perspective.
- **Outcome Event** for meaningful change.
- **Outcome Protocol** if exchange rules become necessary.

## The Core Movement

The simplest form:

> Intent -> Align -> Outcome

The scoped form:

```text
Thread
  Intent
  Align
  Outcome
```

In plain language:

```text
Intent asks: What is being meant or pursued?
Align asks: Do we understand this together, in context?
Outcome asks: What would make this meaningfully true enough?
```

This can produce:

- no visible output
- a clarification
- an artifact
- a process
- a message
- a media moment
- a business workflow
- a learning path
- a social coordination
- a purchase plan
- a creator commission
- a service interaction
- a public contribution
- a private hold
- a refusal
- a pause
- a change in context

Outcome does not mean the world-result has already happened.

Outcome first means Alpha and the relevant authority have aligned on what would be true enough.

After that, Alpha may help through artifacts, processes, actions, media, communication, tools, agents, or silence.

Those later forms should not be collapsed into the core movement.

They should stay understandable around the Thread.

Core rule:

> Every Intent opens a Thread; everything in the movement can point to that Thread.

## Agent-Originated Intent

Agents and sub-agents may form operational Intent.

Examples:

- A researcher agent intends to compare options.
- A boundary agent intends to check whether private memory may be used.
- A process agent intends to resume a waiting task.
- A recovery agent intends to diagnose a failed tool call.
- A context agent intends to inspect whether an old memory is relevant.

This is allowed only if:

- the agent has a scoped identifier
- the Intent is derived or delegated
- the Intent has authority scope
- the Intent has lineage
- the Intent is explainable
- the Intent is interruptible
- the Intent respects the Boundary Layer
- the Intent does not pretend to be human desire

Agent Intent is operational.

Human and group Intent can be sovereign.

## Relationship To Other Alpha Parts

### Life Graph

The Life Graph connects Thread, Intent, Align, Outcome, artifacts, memory representations, processes, boundaries, people, agents, tools, and events.

It should answer:

- Which Thread does this belong to?
- Which Intent led to this Outcome?
- Which part of Align changed this Outcome?
- Which agent initiated this derived Intent?
- Which boundary changed the meaning?
- Which artifact made this Outcome inspectable?
- Which process is currently blocked?
- Which memory representation changed the context?

The Life Graph should not own Thread, Intent, Align, or Outcome.

It connects them.

Thread provides required scope.

Direct relationship claims explain meaning, lineage, responsibility, and current work.

### Boundary Layer

The Boundary Layer governs what Alpha may, may not, and must ask before doing.

Intent can be constrained by boundaries.

Align may require boundary checks.

Outcome can require boundary checks.

High-consequence actions after Outcome should not execute without the right authority.

### Artifact

Artifacts can make parts of Thread, Intent, Align, or Outcome handleable.

Examples:

- Intent Artifact
- Outcome Artifact
- Outcome View
- Thread View
- alignment note
- correction note

Artifacts are handles.

They are not the Thread, the living Intent, the whole act of Aligning, or the whole Outcome.

### Process Layer

The Process Layer tracks dynamic work:

- waiting
- queued
- running
- blocked
- sleeping
- revised
- completed
- abandoned

Align may create processes.

Processes may generate new derived Intent.

Processes may update Outcome State.

Process States may belong to a Thread while serving a specific Outcome.

The Thread gives process state a shared reference.

The Process Layer still owns dynamic work.

### Alignment Layer

The Alignment Layer is different from Align.

Align is the core movement from Intent to Outcome.

The Alignment Layer checks whether Alpha's interpretation, artifact, process, action, or later behavior still fits:

- Intent
- Outcome
- context
- boundaries
- care
- authority
- uncertainty
- user correction
- current reality

The Alignment Layer prevents Alpha from drifting after Outcome is formed.

## Failure Modes

### Intent Collapse

Alpha treats Intent as a literal command.

Risk:

> The deeper meaning disappears.

Correction:

> Preserve source, authority, scope, uncertainty, and context.

### Align Collapse

Alpha skips shared understanding and silently assumes it knows what is meant.

Risk:

> Alpha confidently shapes the wrong thing.

Correction:

> Make assumptions correctable and keep uncertainty visible enough.

### Outcome Collapse

Alpha treats Outcome as execution or task completion.

Risk:

> Alpha finishes something that did not actually match the intent.

Correction:

> Keep Outcome inspectable, editable, and connected to enoughness.

### Thread Collapse

Alpha treats Thread as a project, folder, task list, artifact container, Life Graph, or permission system.

Risk:

> The scope wrapper starts owning responsibilities that belong elsewhere.

Correction:

> Thread owns the organizing reference only; other parts keep their canonical responsibilities.

### Bureaucracy Collapse

Alpha turns every movement into records, forms, approvals, and named objects.

Risk:

> Alpha becomes over-engineered, opaque, or exhausting.

Correction:

> Keep the core movement flat and add structure only where trust, safety, continuation, or control requires it.

### Agent Blur

Alpha attributes all AI-originated work to one vague agent.

Risk:

> Responsibility, source, authority, and rollback become unclear.

Correction:

> Every agent and sub-agent has a scoped identifier and lineage.
